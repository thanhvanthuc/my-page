KeyPulse Dual Mode HUD - Keyboard Visualizer

Privacy Policy

This Privacy Policy describes how KeyPulse Dual Mode HUD - Keyboard Visualizer handles keyboard events, temporary session statistics, webpage access, and locally stored interface preferences. The policy is written specifically for version 1.3.1 of the extension.

Effective date: July 16, 2026Version covered: 1.3.1
Privacy summary

KeyPulse processes supported keyboard events locally so it can update its on-screen keyboard visualizer. It does not maintain a persistent keypress history, does not send typed content to a server, and does not use advertising or analytics. The extension stores only limited interface preferences locally: widget visibility, display mode, and HUD position.

Policy overview

This page explains KeyPulse keyboard-event processing, temporary session statistics, local storage, webpage access, data retention, and privacy safeguards for version 1.3.1.

01

Scope and Purpose of This Policy

This Privacy Policy applies to the KeyPulse Dual Mode HUD - Keyboard Visualizer browser extension, version 1.3.1. It explains the extension's treatment of keyboard events, temporary statistics, locally stored settings, and the webpage access required to display the visual keyboard HUD.

KeyPulse has a narrow and specific purpose: it provides a floating visual keyboard that reacts to supported physical key presses. The extension includes a compact mode and a full keyboard mode and allows users to show, hide, and reposition the HUD.

KeyPulse is not designed as a text-input logger, analytics platform, productivity monitor, advertising service, or cloud-based account product. Its core function is to provide immediate visual feedback inside the browser.

This policy does not govern the privacy practices of the browser itself, the extension marketplace, websites visited by the user, the operating system, or other extensions and applications installed on the device.

02

How the Extension Operates

KeyPulse uses a content script on ordinary HTTP and HTTPS webpages. This allows the extension to place its keyboard visualizer into the webpage and respond to supported keyboard activity while the user is interacting with that page.

The extension's toolbar action can be used to show or hide the visualizer on the current page. The HUD itself can be displayed in compact or full-keyboard mode and can be moved to a preferred screen position.

The visualizer operates locally inside the browser. KeyPulse does not require a developer-operated backend, online account, or remote API to determine which key should be highlighted.

When the current page is reloaded or replaced by navigation, the page's content-script session is recreated. Temporary counters begin again, while locally stored interface preferences can be used to restore the user's chosen HUD configuration.

03

Keyboard Event Processing

KeyPulse listens for browser keydown and keyup events on supported webpages. The extension uses the browser-provided KeyboardEvent.code value to determine which physical key position should be represented in the HUD.

Depending on the key pressed, the extension may highlight the corresponding visual key and may display the recognized key label in compact mode. This can include represented letter keys, number keys, punctuation keys, modifier keys, navigation keys, function keys, Space, Enter, and other supported physical keys.

These events are processed for immediate visual feedback. KeyPulse does not create a persistent chronological record of keyboard activity and does not save a replayable history of every key pressed.

The extension does not intentionally preserve complete typed sentences, search terms, messages, form submissions, or document text. Keyboard events are handled as they occur and are used only for visual state changes and temporary in-memory statistics.

04

Password Fields and Sensitive Input

KeyPulse includes a specific safeguard for standard HTML password fields. When a keydown event originates from an input element with type="password" , the extension ignores that keydown event for the purpose of updating the key visualizer and increasing the temporary press counter.

As a result, ordinary browser-recognized password fields are excluded from the normal keydown visualization path. KeyPulse does not intentionally use those password-field key presses to update the current-key display or temporary page-session statistics.

Some websites may create custom sensitive-input controls that are not technically implemented as standard password inputs. Those design choices are controlled by the website and are outside KeyPulse's control.

Regardless of input type, the reviewed version of KeyPulse does not transmit keyboard events to a developer-controlled backend and does not maintain a persistent remote keypress history.

05

Temporary Press Count and Press Rate

KeyPulse maintains a temporary count of recognized key presses during the active page session. It also calculates an approximate presses-per-minute rate based on the number of recognized presses and the time elapsed since the visualizer session began.

These figures exist only in memory for the current content-script session. They are intended to provide live information inside the HUD and are not written into Chrome's local extension storage.

The temporary count and rate are not uploaded to a server, used for analytics, tied to an account, or combined with activity from other websites.

Reloading the page or navigating to another page creates a new content-script session, causing these temporary statistics to start again.

06

Information KeyPulse Does Not Collect

KeyPulse does not operate as a personal-data collection service. The reviewed implementation does not collect or maintain the following categories for the developer:

  • Names, email addresses, telephone numbers, postal addresses, or account profiles;
  • Passwords, authentication credentials, payment information, or financial details;
  • A persistent transcript or chronological history of keys pressed by the user;
  • Complete typed messages, documents, search terms, or form submissions;
  • A developer-maintained browsing history or database of visited URLs;
  • IP addresses, precise location, advertising identifiers, or demographic profiles;
  • Behavioral analytics, telemetry, session replay, or advertising conversion data;
  • Long-term productivity records based on temporary press counts or press rates.

Because the extension does not maintain a backend for these categories, the developer does not operate a remote keypress database or cloud archive of users' keyboard activity.

07

Local Storage and Saved Preferences

The extension manifest requests the storage permission. In version 1.3.1, Chrome local extension storage is used to preserve a limited set of interface preferences so that the HUD can return in a familiar configuration across browsing sessions.

The stored preferences are limited to:

  • Whether the widget is currently configured to be visible or hidden;
  • Whether the visualizer is using compact mode or full-keyboard mode;
  • The horizontal and vertical position used to restore the floating HUD.
What local storage is not used forThe extension does not use local storage to save typed text, a keypress history, the temporary total-press counter, the temporary presses-per-minute value, webpage content, or form-field contents.

These preferences remain in the browser's local extension storage. The reviewed implementation does not include a developer-controlled synchronization service that receives this local preference data.

Users can remove these preferences by clearing extension data or uninstalling KeyPulse.

08

Website Access

KeyPulse declares content-script access for ordinary HTTP and HTTPS webpages. This is necessary because the visualizer is intended to appear directly on webpages and respond to keyboard events that occur while the user interacts with those pages.

This access allows the extension to insert its HUD interface and attach the event listeners required for the visualizer. It does not mean that KeyPulse maintains a browsing-history service or sends page content to the developer.

The reviewed code does not use this access to capture screenshots, build a database of webpage text, collect form submissions, inspect private messages for analytics, or send webpage content to a remote server.

Some browser-controlled and internal pages restrict content scripts. On those pages, the extension may be unable to display or toggle the HUD because the browser does not allow the required script to run.

09

Network Requests and Remote Code

The reviewed version of KeyPulse does not use remote code for its keyboard visualizer. The executable functionality required to create the HUD, process supported keyboard events, update visual states, calculate temporary statistics, and store interface preferences is included in the installed extension package.

The implementation does not contain a developer-configured analytics pipeline or backend service used to transmit keyboard activity. There is no server endpoint used to upload key presses, typed text, temporary press counts, or presses-per-minute values.

The browser and extension marketplace may independently perform network operations for installation, updates, malware checks, account services, or security. Those platform activities are outside KeyPulse's control and are governed by the relevant platform's own privacy practices.

10

Advertising, Analytics, Tracking, and Profiling

KeyPulse does not display advertising and does not include advertising SDKs, tracking pixels, session replay systems, browser-fingerprinting code, or behavioral analytics for commercial profiling.

Keyboard activity processed for the visualizer is not used to infer purchasing intent, health information, financial characteristics, employment performance, personal interests, or other unrelated traits.

The extension is not designed to convert keyboard behavior into an advertising profile and does not sell or rent keyboard activity or local HUD settings to advertisers, data brokers, or analytics providers.

11

Data Sharing and Disclosure

KeyPulse does not intentionally share keyboard events, typed content, temporary press counts, press-rate statistics, or locally stored HUD preferences with third parties.

Because the extension does not maintain a developer-operated store of keyboard activity, there is ordinarily no remote keyboard dataset available for commercial sharing or disclosure.

Local extension settings remain within the browser environment unless the user or the browser itself removes them.

12

Data Retention

Individual keyboard events are processed as they occur and are not retained by KeyPulse as a persistent chronological history.

The temporary total-press count and presses-per-minute rate remain only for the active page session and reset when the page session is recreated.

The only information intentionally retained across sessions is the limited interface configuration stored in Chrome local extension storage: widget visibility, visualizer mode, and saved HUD position.

These settings remain until they are changed, cleared with extension data, or removed when the extension is uninstalled.

13

Security and Privacy Design

KeyPulse limits privacy risk by performing its primary keyboard visualization locally and by avoiding a remote data pipeline for keyboard activity.

The extension ignores keydown events from standard password input fields. It also clears active visual key states when the browser window loses focus or when the document becomes hidden, helping prevent stale key highlights from remaining on screen after the user's context changes.

No software can eliminate every possible privacy or security risk, especially when its declared function requires receiving browser keyboard events. KeyPulse therefore follows a purpose-limited design: process the event for immediate visualization, avoid building a persistent key history, avoid transmitting keyboard activity, and retain only the minimum interface settings needed for usability.

14

Changes to This Privacy Policy

This Privacy Policy may be revised if the extension's implementation changes. Examples that would require a meaningful policy update include adding new permissions, introducing a backend service, changing how keyboard events are stored, adding analytics, expanding site access, or retaining new categories of information.

Future versions should be reviewed against the actual source code and manifest rather than assuming that the behavior described for version 1.3.1 remains unchanged.

Any updated policy should include a revised effective date and accurately explain new data flows before those practices are represented as part of the extension.

15

Contact

Questions about this Privacy Policy or the privacy behavior of KeyPulse Dual Mode HUD - Keyboard Visualizer may be directed to the support contact published with the extension's official store listing.

Users do not need to provide passwords, private text, or other sensitive content when contacting support about privacy or extension behavior.
